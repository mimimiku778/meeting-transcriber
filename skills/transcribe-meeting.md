# /transcribe-meeting スキル

会議動画の文字起こし・議事録作成・検索を行う対話的ワークフロー。

設計の柱（このスキルの背骨。常に守る）:
- **ゼロお膳立て**: ユーザーは動画を渡すだけ。案件・議題・話者・フレーム位置・取捨は全部こちらが
  自動で判定する。事前に選ばせる質問・設定ファイルの手作業はさせない。
- **修正ベース学習**: 先に確認して止めず、まず最善推定で議事録(成果物)まで出す。ユーザーが文面を
  直したら、その修正を案件ストアへ書き戻して育てる（声紋と同じ『育てる』仕組みの案件版）。
- **成果物段階で話者を正す**: 1次の文字起こし(txt)の精度には拘らない。diarizationの過分割
  （同一人物が発話者5/6に分裂）・過少分割（1ラベルに別人混在）は、声紋ヒントを使って
  **議事録を書く段で**解決する。
- **声紋と案件ストアは同一 slug で連結**: `~/.claude/voiceprints/<slug>.json`（声）と
  `~/.claude/meeting-contexts/<slug>.yaml`（組織図・用語・取捨の好み）。どちらもローカルのみ。

## 使い方

```
/transcribe-meeting [動画パス]
```

---

## 最初にやること（必須）

**動画パスが渡されている場合は、下のモード選択を出さずに「新規・自動フロー」をそのまま開始してよい**
（ゼロお膳立ての原則。判定は内部で自動）。動画が無い／意図が曖昧なときだけ、次を出して選ばせる:

```
会議議事録アシスタントです。何をしますか？

A. 新しい動画を文字起こしする（案件・話者は自動判定）
B. 既存の議事録を編集・更新する
C. 過去の議事録を検索・まとめ生成する

────────────────────────────────
📊 Whisperモデル（ハルシネーション抑制・用語集注入・正規化は全モデル共通）:
  large-v3-turbo  large並み精度をmedium並み速度で（~1.6GB）← 既定・推奨
  large-v3 最高精度・低速 / medium 軽量退避 / small 試し用
🎙️ 話者識別: speakrs（Apple Silicon・最速）← 既定 / pyannote（CPU低速）
📇 案件は自動判定: ~/.claude/meeting-contexts/<slug>.yaml を signals で照合。
   無ければ新規案件として進め、あなたの議事録修正から自動で育てる（手作業不要）。
💡 別ターミナル: transcribe --watch（進行監視） / transcribe --kill（強制終了&MCP再起動）
────────────────────────────────
```

## 利用可能なMCPツール

文字起こし系:
- `transcribe_meeting`: 動画→文字起こし（話者識別付き）。`project` を渡すと案件ストアを
  コンテキスト＋声紋として自動適用し、話者同一性ヒントを `<transcript>_speakers.json` に出力。
- `extract_video_frame`: 1枚だけ抽出＋OCR。`output_dir` 指定で `frames/` に永続保存。
- `extract_video_frames`（**複数枚は必ずこちら**）: タイムスタンプ配列を渡すと動画を1回だけ開いて
  まとめて抽出し、OCRを並列実行する。1枚ずつ呼ぶより大幅に速い。
- **抽出後はどちらも必ず各 `image_path` を `Read` で視覚確認**（OCRが拾えないUI・図表・青枠＝発話中の話者を見る）。
- `read_transcript` / `update_speaker_names` / `finalize_meeting_files`: 読込・話者名置換・整理。

案件＆話者の自動化系（このスキルの中核）:
- `identify_project`: 動画パス・フレームOCR語・声紋一致から**どの案件かを自動判定**。候補を確信度順に返す。
- `resolve_speakers`: 動画の**話者同一性ヒント**（声紋クラスタ類似度＝過分割の統合候補／混在検出＝
  過少分割／登録済み声紋による区間単位の実名照合）を算出し `<video>_speakers.json` に保存。
  ※ `transcribe_meeting` に `project` を渡せば同じサイドカーが自動生成されるので、通常は単独呼び出し不要。
- `upsert_project_context`: ユーザーが議事録で確定・修正した情報を案件ストアへマージして育てる
  （**修正ベース学習の書き戻し**）。組織図/ロスター/用語/帰属ルール/取捨の好み(minutes_preferences)/
  判定シグナル等を部分更新。人間確定が常に優先。
- `list_projects`: 登録済み案件の一覧。
- `enroll_voiceprints`: 確定した「発話者N→実名」から声紋を登録/更新。**議事録を出した後の最後に回す**
  （次回以降の自動実名化のための投資で、今回の議事録には不要。クリティカルパスに入れない）。

DV## 新規文字起こしモード（自動フロー）

### Step 0: 案件の自動判定（ゼロお膳立ての入口）
1. `identify_project(video_path)` を呼ぶ（パスのディレクトリ/ファイル名で軽く照合）。
2. さらに冒頭フレーム（0〜30秒で参加者パネル・タイトルが映る所）を `extract_video_frame` で取り、
   `Read` で視覚確認。OCR語・組織名を `identify_project(video_path, ocr_terms=[...])` に渡して再判定。
3. 判定結果の扱い（**曖昧でも止めない**）:
   - 明確な最上位候補があれば、その `slug` を採用して進む。
   - 弱い／割れている／該当なしでも止めない。**最善推定の slug**（既存に寄せるか、内容から作った
     新規 slug。例: ディレクトリ名や主組織から `myteam` のような短い語）で進む。
   - 新規案件のときは slug を内部で決め、以後の修正からストアを育てる（ユーザーには確認で煩わせない）。

### Step 1: モデル選択（任意・既定で即進行可）
既定（large-v3-turbo / speakrs）で良ければ確認不要で進める。ユーザーが精度要求を出したときだけ
large-v3 等に上げる。

### Step 2: 文字起こし（案件・声紋・話者ヒントを一括適用）
`transcribe_meeting(video_path, project=<slug>)` を呼ぶ。これで:
- 案件ストアの固有名詞をASRに注入＋決定的正規化
- 同一 slug の声紋で `発話者N` を自動実名化（未登録・低信頼は `発話者N` のまま）
- **話者同一性ヒントを `<transcript>_speakers.json` に自動出力**（統合候補・混在警告・区間照合）

### Step 3: ファイル整理
`read_transcript` で内容把握 → 会議タイトルを生成（10〜20字）→ `finalize_meeting_files` で
タイトル付きディレクトリへ整理。検出話者数を報告。

### Step 4: 議事録作成（成果物段階で話者を解決し、取捨を学習適用）

**Step 4-1: 話者同一性の解決（1次結果は直さず、議事録側で正す）**
`<transcript>_speakers.json` を `Read` し、次を**議事録の話者帰属**に反映する:
- `merge_suggestions`（**過分割＝同一人物の可能性**）: 例「発話者5+発話者6 類似度0.91」→ 議事録では
  同一人物として一貫表記する。フレームの青枠（発話中）で同一人物か裏取りする。
- `mixed_warnings`（**過少分割＝別人混在の可能性**）: そのラベルは1人に見えて複数人かもしれない。
  発言ごとに文脈・フレーム・`segment_relabel` で振り分け、混在のまま1人にしない。
- `segment_relabel`（**区間単位の声紋照合**・登録済み声紋がある場合）: 「発話者3 130〜148s→鈴木」等。
  その時刻の文字起こし行を該当実名として扱う（音声時刻↔文字時刻は対応済み）。
- `identified`（声紋で実名化済み）: そのまま採用。
確定した「発話者N→実名（会社）」の対応（**多対一を許す**＝5も6も同一人物なら同じ実名）で議事録を書く。

**Step 4-2: 話者名・帰属のフレーム裏取り**
参加者パネル（冒頭）と、アクティブスピーカー（青枠）が映るコマを `extract_video_frame`＋`Read` で確認し、
氏名↔会社↔話者を確定する。確定できた対応は `update_speaker_names` で文字起こしにも反映してよい
（議事録の主たる解決は Step 4-1 の声紋ヒント＋ここのフレームで行う）。

**Step 4-3: 画面共有フレームの自動収集（位置はユーザーに指定させない・必ずバッチ）**
文字起こしを分析し、画面共有・資料説明・話題転換・具体数値の箇所を自動で選ぶ。**候補時刻は全部まとめて
1回の `extract_video_frames(timestamps_seconds=[...])` で抽出する**（1枚ずつ呼ばない＝速度のため必須）。
- 「画面を共有します」等の宣言は**直後にも資料が出る**ため、宣言時刻＋数秒〜十数秒ずらした候補も同じ配列に
  入れて撮り、`Read` で見比べて**情報量の多い1枚**を採用する（タイミングのズレを自動で吸収）。
- 目安: 30分=5〜6枚 / 1時間=10〜12枚 / 画面共有が多ければ追加。Step 4-2 の参加者パネル・青枠の時刻も
  この配列にまとめると動画オープンが1回で済む。各フレームは必ず `Read` で視覚確認。

**Step 4-4: 議事録生成（取捨の学習を適用・根拠必須・自己検証）**
案件ストアの組織図/ロスター/帰属ルール、そして **`minutes_preferences`（この案件で学習済みの
「何を残し何を書かないか」）** に従って書く。

書き方の原則（常に適用する品質ルール）:
- **結論ファースト**: 各セクションは結論（決定・アクション）を先に置き、根拠・詳細は入れ子で書く。
- **冗長カット**: 相槌・段取り・本題でない雑談など流れ的にどうでもいいやり取りは省く。必要な情報だけ
  見やすく、重要な箇所だけ細部を残す。
- **「要確認／持ち帰り」は確定か曖昧かを切り分ける**: 会議で明示的にコミットされた確認・持ち帰りは
  確定タスクとして断定形で書き、曖昧なまま流れた・未決のものは課題として「未決」と分かる形で書く。
  自分の推測で「要確認」を足さない。
- **そこにしかない事象（システムの状態・不具合・制約など）は残す**。一方、既にアクション項目に反映済みで
  自明な指摘は省く。
- **温度感・機微を残すときは含意付きで**: 形容（率直・こだわり等）だけの宙に浮いた記述にしない。
  「誰が誰に何を求めているか／次の判断にどう効くか」が分かる形で書く（含意が無いなら省く）。

生成手順:
1. 議論要約＋フレーム視覚情報（資料タイトル/数値/宛先）を統合。
2. 各論点を相互排他に分類: `decision`（明示合意あり）/`action_item`（今後の作業）/
   `open_issue`（未決）/`parking_lot`（保留）。合意引用の無い「決定」は open_issue に降格（捏造禁止）。
3. action_item と一部 decision に **owner（組織ストアの id か UNKNOWN）** を付与。各項目に
   `owner` / `evidence_quote`（原文逐語）/ `speaker` / `timestamp(MM:SS)` / `confidence` を付ける。
   **引用が原文に無い項目は作らない。**
4. 自己検証(CoVe): 引用の逐語存在・分類定義・owner整合を点検。不整合は降格 or UNKNOWN。
5. ユーザー確認は **owner=UNKNOWN と confidence=low の項目だけ**に絞る（全件レビューさせない）。
6. `finalize_meeting_files` が返したパスに保存。決定/宿題（担当・期限）/課題のセクションと根拠引用を含める。

推奨フォーマット:
```
## 決定事項
- [D1] （決定内容）（根拠: 山田 02:44「（合意の逐語引用）」→合意）

## 宿題（アクションアイテム）
| # | 内容 | 担当 | 期限 | 根拠（話者/時刻/引用） | 確度 |
|---|------|------|------|----------------------|------|
| A1 | （作業内容） | A社 | 期限 | 山田 04:13「（逐語引用）」 | high |

## 課題（未決・要検討）
- [I1] （未決事項）（要確認）
```

### Step 5: 修正ベース学習（ユーザーの文面修正を案件ストアへ書き戻す）
議事録を提示し、ユーザーが直した内容から学習材料を抽出して `upsert_project_context(slug, updates)` で
書き戻す（**ユーザーに別途お膳立てさせない。文面への修正＝学習入力**）。拾う対象:
- 話者の訂正（「5と6は同じ山田」「この発言は佐藤(A社)」）→ `speaker_roster`（name/company）。
- 用語・社名・誤認識の訂正 → `glossary`（term＋aliases）／必要なら `normalization.deterministic`。
- 「これは残す/これは書かない」の指示や、消された/足された傾向 → `minutes_preferences`
  （rule＋polarity=keep/drop＋source=explicit|edit-diff）。
- 案件判定が外れていたら、正しい slug の `signals`（dir_keywords/org_terms）を補強。
- 組織の役割・帰属の訂正 → `organization`／`attribution_rules`。

**声紋登録は最後に（後回し）**: 議事録が固まり、フレームで高信頼に確定できた「発話者N→実名」だけを
`enroll_voiceprints(video_path, voiceprint_profile=<slug>, speaker_mapping=...)` で登録する。
過統合・低信頼の話者は誤学習防止のため据え置く。重い処理なので議事録提出後に回し、急ぐときは省略可
（次回の自動実名化が少し弱くなるだけで、今回の成果物には影響しない）。

## 既存議事録編集モード

`*_transcript_*.txt` / `*_minutes_*.md` / `frames/` / `*_speakers.json` を Glob で把握し、
A.修正・更新 / B.新バージョン生成 / C.誤字修正 / D.フレーム追加抽出 / E.確認のみ を選ばせる。
編集で確定した訂正は、可能なら Step 5 と同様に `upsert_project_context` で案件ストアへ反映して育てる。

---

## 検索モード

指定ディレクトリ配下の `*_transcript_*.txt` / `*_minutes_*.md` を Grep。キーワードと、タイムスタンプ
`(MM:SS)` から動画の何分何秒かを示す。複数議事録からのまとめ生成は目的/形式を確認して `Write` で保存。

## 重要事項
- 常に日本語で応答。エラーは原因と対処を示す。
- **ゼロお膳立て**: 案件・議題・話者・フレーム位置を自動で決める。事前確認の質問・手作業設定をさせない。
- **成果物段階で話者を正す**: 1次txtは直さなくてよい。過分割は声紋類似度で統合、過少分割は混在警告＋
  区間照合＋フレームで議事録上で分離。混在のまま1人にしない。
- **修正ベース学習**: ユーザーの議事録修正を `upsert_project_context` で案件ストアへ書き戻して育てる。
- **帰属はフレーム由来の確定事実を最優先**: 略号を額面で読み替えない。確証が無ければ UNKNOWN（捏造禁止）。
- **宿題・決定には必ず根拠（話者・時刻・逐語引用）**。引用が原文に無い項目は作らない。
- **フレームは必ず `Read` で視覚確認**。位置はユーザーに指定させず、複数候補から情報量の多い1枚を選ぶ。
- **議事録は必ずファイル出力**（ターミナルに全文を出さずパスを示す）。
- **声紋登録(enroll)は議事録の後に回す**（重い・今回の成果物には不要）。急ぐときは省略可。
- **速度優先・並列化**: フレームは必ず `extract_video_frames` で一括＋並列OCR。独立した抽出・解析
  （フレーム抽出と話者ヒントの確認など）は1メッセージ内で複数ツールを同時に呼んで並列実行する。
  音声抽出は動画stem単位でキャッシュ再利用される（transcribe/enroll/resolveで重複抽出しない）。
